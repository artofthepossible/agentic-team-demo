## Backend Specifications

### Data Model
type Account struct {
    ID        string    `json:"id"`         // UUID v4
    Name      string    `json:"name"`
    Email     string    `json:"email"`
    Balance   float64   `json:"balance"`    // stored as INTEGER cents in SQLite
    CreatedAt time.Time `json:"created_at"`
}

type Transaction struct {
    ID        string          `json:"id"`
    AccountID string          `json:"account_id"`
    Type      TransactionType `json:"type"`   // "deposit" | "withdrawal"
    Amount    float64         `json:"amount"`
    Note      string          `json:"note"`
    CreatedAt time.Time       `json:"created_at"`
}

### API Endpoints
POST   /api/accounts                    → Create account (name, email, initial_deposit)
GET    /api/accounts/:id                → Get account details + balance
GET    /api/accounts/:id/transactions   → List transaction history (paginated)
POST   /api/accounts/:id/deposit        → Deposit funds (amount, note)
POST   /api/accounts/:id/withdraw       → Withdraw funds (amount, note)
GET    /api/health                      → Health check

### Business Rules
- All monetary values stored as integer cents (int64) in SQLite; converted to float64 in API
- Withdrawals must be rejected if balance < amount (return 422 with clear error message)
- Amount must be > 0 for all transactions (validate on both backend and frontend)
- Email must be unique; return 409 Conflict if duplicate
- Use database transactions (BEGIN/COMMIT) for deposit and withdraw operations
- All errors returned as JSON: {"error": "message", "code": "ERROR_CODE"}

### Persistence
- Use SQLite via mattn/go-sqlite3 with WAL mode enabled
- Run schema migrations on startup (CREATE TABLE IF NOT EXISTS)
- Store balance as INTEGER (cents) to avoid floating point issues
- Index accounts by email for uniqueness checks

### Middleware
- CORS: Allow http://localhost:5173 (Vite dev server) in development
- Request logger: method, path, status, latency
- Panic recovery: return 500 JSON instead of crashing

## Frontend Specifications

### Stack
- React 18 with Vite
- Tailwind CSS for styling
- Axios for HTTP calls
- React Query (@tanstack/react-query) for data fetching, caching, and mutations

### Design System
Visual identity: "vault modernism" — precision banking meets warm accessibility.

Palette:
- Background:  #0F1117  (near-black, not pure black)
- Surface:     #1A1D27  (card background)
- Border:      #2A2D3E  (subtle dividers)
- Accent:      #4F8EF7  (electric blue — primary actions)
- Success:     #34D399  (deposits, positive balance)
- Danger:      #F87171  (withdrawals, errors)
- Text:        #E8EAF0  (primary), #8B8FA8 (muted)

Typography:
- Display: "Inter" (weights 700/800 for balance figures)
- Body: "Inter" (400/500 for labels and UI)
- Monospace: "JetBrains Mono" for account IDs and amounts

Signature element: The balance display uses a large tabular-numeric counter with
a subtle animated gradient shimmer on load — the number is the hero of the UI.

### Pages / Views
1. Home: Search or create account (centered, minimal)
2. Account Dashboard:
   - Large balance display (the "vault" hero number)
   - Deposit / Withdraw forms side by side
   - Scrollable transaction history with type icons + timestamps
3. Create Account Modal: name, email, optional initial deposit

### UX Requirements
- Optimistic UI for deposit/withdraw (show pending state)
- Invalidate and refetch balance + history after every transaction
- Form validation before API call (amount > 0, required fields)
- Display backend error messages inline (not just console)
- Responsive: works on mobile (single-column) and desktop (two-column)
- Empty state for transaction history: "No transactions yet — make your first deposit"

## Docker & Container Specifications

### Dockerfile (multi-stage, single image serves everything)

Use a 3-stage build:

Stage 1 — frontend builder (node:20-alpine):
  - WORKDIR /frontend
  - Copy frontend/package.json and frontend/package-lock.json
  - RUN npm ci                          # reproducible install
  - Copy rest of frontend/
  - RUN npm run build                   # outputs to /frontend/dist

Stage 2 — backend builder (golang:1.22-alpine):
  - Install gcc and musl-dev (required for mattn/go-sqlite3 CGO)
  - WORKDIR /app
  - Copy go.mod and go.sum first, then RUN go mod download
  - Copy remaining Go source files
  - Copy --from=frontend-builder /frontend/dist ./frontend/dist
  - ENV CGO_ENABLED=1
  - RUN go build -o banking-server .    # statically linked where possible

Stage 3 — runtime (alpine:3.19):
  - RUN apk add --no-cache ca-certificates sqlite
  - Create non-root user: adduser -D appuser
  - WORKDIR /app
  - Copy --from=backend-builder /app/banking-server .
  - Create /data directory for SQLite, chown to appuser
  - USER appuser
  - EXPOSE 8080
  - ENV DB_PATH=/data/banking.db PORT=8080
  - ENTRYPOINT ["./banking-server"]

The Go server must serve the React build as static files from ./frontend/dist
at the root path (/), and serve /api/* routes from the Go handlers.
Use http.FileServer with a fallback to index.html for client-side routing.

### .dockerignore
Exclude the following to keep build context lean and avoid cache poisoning:
  frontend/node_modules
  frontend/.vite
  frontend/dist
  **/*.db
  **/*.db-shm
  **/*.db-wal
  .git
  .gitignore
  README.md
  **/*_test.go
  **/testdata/

### docker-compose.yml
Version: "3.9"

Services:

  app:
    build:
      context: .
      dockerfile: Dockerfile
    container_name: banking-app
    ports:
      - "8080:8080"
    volumes:
      - banking_data:/data          # persist SQLite database across restarts
    environment:
      - PORT=8080
      - DB_PATH=/data/banking.db
      - ENV=production
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "wget", "-qO-", "http://localhost:8080/api/health"]
      interval: 30s
      timeout: 5s
      retries: 3
      start_period: 10s

volumes:
  banking_data:
    driver: local

### Container Design Decisions
- Single image: Go server statically serves the compiled React app — no nginx sidecar needed
- Named volume banking_data: SQLite file survives container restarts and rebuilds
- Non-root user (appuser): principle of least privilege inside the container
- Health check on /api/health: Docker will mark the container unhealthy if the Go server fails
- CGO_ENABLED=1 required: mattn/go-sqlite3 uses CGO; alpine builder includes musl-dev
- Build cache optimisation: go.mod/go.sum and package.json copied before source so
  dependency layers are cached and only invalidated when deps actually change

## Quality Standards
- All handlers must return JSON (never plain text)
- No global mutable state in Go; pass dependencies via struct receivers
- React components should be small and focused (< 150 lines each)
- Use environment variable PORT (default 8080) for the Go server port
- Docker image must build cleanly with: docker compose up --build
- docker compose down followed by docker compose up must preserve all account data
- The compiled Docker image should be under 50MB
